# 프로젝트 기획서

## 1. 문제 정의

### 1.1 해결하려는 문제

파트타임, 외주, 블로그, 앱 서비스 등 여러 부수입원을 운영하는 사람들은 공통적으로 이런 어려움을 겪습니다.

- **수입원마다 성격이 너무 다르다**: 외주는 건당 정산, 블로그는 월 정산, 앱은 불규칙 수익 등 — 각각의 수익 주기와 지출 구조가 달라서 하나로 묶어 관리하기 어렵습니다.
- **수익성을 판단할 기준이 없다**: "외주를 계속 하는 게 맞나?", "블로그에 쓰는 시간 대비 수익이 괜찮은 건가?" 같은 질문에 답할 근거가 없습니다.
- **기존 도구로는 부족하다**: 엑셀이나 가계부 앱은 단순 입출금 기록에 그치고, '프로젝트별 수익성 비교' 같은 분석은 제공하지 않습니다.

### 1.2 이 문제를 선택한 이유

단순히 "얼마 벌었다"를 기록하는 것을 넘어서, **내가 투자하는 시간과 비용 대비 어떤 활동이 가장 효율적인지** 판단하고 싶었습니다. 돈이 안 되는 활동은 줄이고, 효율 좋은 활동에 집중하기 위한 도구가 필요했습니다.

### 1.3 기능 구분

#### 핵심 기능

- **수입원 등록 및 관리**: 각 부수입 활동을 수입원 단위로 등록하고, 유형별로 분류(프리랜서/프로젝트/수동소득 등).
- **거래 기록**: 수입원별로 수익과 지출을 날짜별로 기록. 투입 시간도 함께 기록 가능.
- **통합 대시보드**: 전체 수입원의 이번 달 수익, 지출, 순수익을 요약하고, 지난달 대비 증감을 한눈에 확인.
- **시간당 수익 계산**: 투입한 시간 대비 실제로 얼마를 벌었는지 자동 계산. 최저시급과 비교해서 효율을 판단.
- **투자 대비 수익률 계산**: 들어간 비용 대비 얼마나 남았는지 백분율로 표시. 돈을 쓸 가치가 있는 활동인지 판단.
- **수입원별 비중 시각화**: 전체 수익에서 각 수입원이 차지하는 비중을 차트로 표시. 특정 수입원에 지나치게 의존하고 있는지 확인.

#### 부가 기능

- **거래 수정**: 잘못 입력한 거래를 삭제 후 재입력 없이 바로 수정.
- **거래 필터**: 수익만, 지출만 골라서 보기.
- **수입원 활성/비활성 전환**: 현재 진행 중이 아닌 수입원을 비활성 처리.
- **수입원 간 효율 비교**: 대시보드에서 어떤 수입원의 시간당 수익이 가장 높은지, 투자 대비 수익률이 가장 좋은지 비교.

### 1.4 대상 사용자

2개 이상의 부수입을 운영하는 프리랜서, 개발자, 크리에이터.

#### 사용 시나리오

1. 퇴근 후 사이드 프로젝트 개발에 2시간을 투입하고, 시간과 함께 기록한다.
2. 외주 건으로 입금된 50만 원을 수익으로 기록한다.
3. 월말에 대시보드에서 확인: 외주는 시간당 5만 원의 높은 효율, 사이드 프로젝트는 아직 적자.
4. 데이터를 보고 "당분간 외주 비중을 유지하면서 사이드 프로젝트의 수익 전환을 지켜보자"는 판단을 내린다.

### 1.5 예외 상황 및 처리 방안

- **투입 시간이 0인 수입원**: 수동소득(블로그 광고 등)처럼 시간 투입이 없는 경우, 시간당 수익은 "해당 없음"으로 표시.
- **지출만 있고 수익이 없는 수입원**: 아직 수익이 발생하지 않은 초기 프로젝트. 투자 대비 수익률이 마이너스로 표시되며, 현재까지의 투자 금액을 명확히 보여줌.
- **거래가 하나도 없는 수입원**: 빈 상태 화면에서 첫 거래 추가를 안내.

## 2. 개선 과정 기록

초기 아이디어에서 현재의 서비스로 발전하기까지의 과정입니다.

### 2.1 초기 설계와 변경 사항

#### 처음 구상

- 수입원 이름과 금액만 보여주는 단순한 카드 리스트.
- "수입 얼마", "지출 얼마"만 입력하는 단순 기록.
- 기본 제공되는 UI 라이브러리 테마를 그대로 사용.

#### 변경 및 개선

1. **데이터 구조 구체화**:
   - 처음에는 단순히 금액의 +, -만 기록했으나, 이것만으로는 "외주 수익"과 "앱 수익"의 성격 차이를 구분할 수 없었습니다.
   - 수입원에 유형 분류를 도입하고, 거래마다 수익/지출 구분, 날짜, 설명을 기록하도록 확장했습니다.
2. **화면 디자인 전면 교체**:
   - 초기 버전은 알록달록한 색상으로 일반 가계부 앱처럼 보였습니다. "전문적인 분석 도구"라는 느낌을 주기 위해, 흑백 톤 중심의 깔끔한 디자인으로 전면 교체했습니다.
3. **화면 구조 변경**:
   - 단일 페이지 스크롤 방식에서, 왼쪽 사이드바로 메뉴를 배치하는 구조로 변경했습니다. 대시보드와 수입원 관리 화면 사이를 빠르게 오갈 수 있게 되었습니다.
4. **차트 색상 개선**:
   - 흑백 디자인에 맞춰 차트까지 회색으로만 표시했더니, 데이터 간 구분이 전혀 되지 않았습니다. 화면 전체는 흑백을 유지하되, 차트에만 구분 가능한 색상 팔레트를 적용했습니다.

### 2.2 막혔던 부분과 해결 방법

- **문제**: 서로 다른 단위의 수입원(월급, 건당 외주비, 광고 수익)을 하나의 그래프에 넣으니 비교가 어려웠습니다.
- **해결**: 절대 금액 비교 대신, 전월 대비 증감률이나 전체 수익 중 비중 같은 상대적인 수치를 우선 보여주는 방식으로 전환했습니다.

- **문제**: 차트의 x축 레이블이 "2026-01"처럼 길어서 미니 차트에서 잘려 보이고, 막대와 레이블 위치가 어긋났습니다.
- **해결**: 레이블을 "1월" 형태로 축약하고, 숨겨진 y축이 차지하는 공간을 제거하여 정렬 문제를 근본적으로 해결했습니다.

### 2.3 우선순위 및 시간 배분

- **1단계 (기반 구축)**: 수입원과 거래의 등록/조회/삭제, 기본 대시보드 차트 — 핵심 데이터 흐름 완성에 집중.
- **2단계 (분석 기능)**: 시간당 수익, 투자 대비 수익률 계산, 거래 수정/필터 등 — "기록 도구"에서 "분석 도구"로의 전환. 현재 진행 중.

### 2.4 계산 로직 개선 (정교화)

초기에는 단순 합계를 사용했으나, **"실질적인 효율"**을 판단하기 위해 계산 방식을 정교하게 다듬었습니다.

- **문제**: 시간당 수익을 `총 매출 / 투입 시간`으로 계산하니, 지출이 많은 고비용 프로젝트의 효율이 과대평가되는 문제가 있었습니다. (예: 매출 100만 원, 지출 90만 원인 경우 시급이 높게 잡힘)
- **논의**: ROI(투자 대비 수익률)에 '시간'을 비용으로 환산해 넣을지 고민했으나, ROI는 '자본 투자 효율'을 보는 지표로 남겨두기로 결정했습니다.
- **해결 (시간당 순수익 도입)**:
  - **기존**: `총 매출 / 총 시간` (단순 매출 생산성)
  - **변경**: `(총 매출 - 총 지출) / 총 시간` = **`순수익 / 총 시간`**
  - **효과**: 비용을 뺀 **"진짜 내 인건비"**를 정확하게 파악할 수 있게 되었습니다.

### 2.5 시각화 및 인사이트 고도화 (Visualization & Insight Enhancement)

데이터가 쌓일수록 단순한 숫자 나열만으로는 직관적인 판단이 어려워졌습니다. "그래서 뭐가 제일 효자 종목인데?"라는 질문에 즉각적으로 답하기 위해 시각화 방식을 개선했습니다.

- **문제**: 수입원 목록만으로는 각 프로젝트의 '규모감'과 '성장세'를 한눈에 비교하기 어려움.
- **해결**:
  - **효율성 트리맵(Efficiency Treemap)**: 수입원의 크기(사각형 면적)로 매출 비중을, 색상 농도로 수익률을 표현하여 "덩치 큰 사업"과 "알짜 사업"을 직관적으로 구분.
  - **인사이트 카드(Insight Cards)**: 사용자가 직접 계산할 필요 없이 "최고 수익", "최고 효율(Hourly Rate)", "최고 성장" 수입원을 자동으로 하이라이트.
  - **추세선(Trend Line)**: 월별 들쭉날쭉한 수입 변동 속에서 장기적인 성장 흐름(Moving Average)을 파악할 수 있도록 보조 지표 추가.

## 3. AI 활용 기록

### 3.1 제공한 맥락

- **프로젝트 규칙 문서**: 한국어 사용, 파일 크기 제한(200줄), 컴포넌트 분리 원칙 등 구체적인 제약 조건을 사전에 정리하여 공유.
- **디자인 방향**: "화려한 것보다 정보가 잘 읽히는 디자인"이라는 방향성을 명확히 제시.
- **기술 스택 한정**: 사용할 프레임워크와 라이브러리를 미리 정해두어, 불필요한 도구 제안을 방지.

### 3.2 검증 및 수정

- 자동 생성된 코드가 200줄을 넘으면 "분리하라"고 지시하여 규칙 준수 유도.
- 수익 계산 로직에서 마이너스 수익 같은 경계 상황을 직접 제시하며 로직 보완.
- 디자인 수정 시 "그림자를 줄이고 테두리 위주로"처럼 구체적으로 요청.

### 3.3 협업에서 발견한 점

- **장점**: 반복적인 구조 코드(컨트롤러, 서비스 기본 틀 등) 작성이 빨라졌습니다.
- **어려웠던 점**: "왜 외주에는 시간당 수익이 중요하고 블로그에는 투자 대비 수익률이 중요한지" 같은 비즈니스 맥락을 한 번에 전달하기 어려웠습니다.
- **대응**: 기획 의도를 별도 문서로 정리하여 지속적으로 공유하고, 단계별로 설명하는 방식을 채택했습니다.
