## 2026-02-11: Phase 2 — 서비스 고도화 기획

시드 데이터(7개 수입원, 100건 거래 내역) 투입 이후, 서비스를 "실제로 의사결정에 도움이 되는 도구"로 발전시키기 위한 4가지 주요 작업을 기획했습니다.

### 1. 대시보드 고도화

현재 대시보드는 기본적인 요약 카드(순수익/총수익/총지출) + 월별 바 차트 + 수익 순위만 제공합니다.

#### A. 전월 대비 증감 표시

- 각 통계 카드에 **"전월 대비 +15%"** 같은 변동 지표를 추가합니다.
- 사용자가 현재 추세가 상승인지 하락인지 **한눈에 판단** 가능하게 합니다.

#### B. 수입원별 포트폴리오 파이 차트

- 전체 수익의 **수입원별 비중**을 시각화합니다 (예: 외주 60%, 블로그 15%, 강의 25%).
- 소득이 특정 수입원에 **과도하게 집중되어 있는지** 파악할 수 있게 합니다.
- 파이 차트가 기존 차트와 함께 대시보드 하단에 배치됩니다.

### 2. 수입원 상세 페이지 강화

#### A. 월별 수익/지출 미니 차트

- 해당 수입원의 **월별 추이를 시각적으로** 표현합니다.
- 수입원 상세 페이지 상단에 요약 카드 아래에 배치합니다.

#### B. 거래 필터/검색

- 수익(+) / 지출(-) **타입별 필터링** 기능을 추가합니다.
- 거래 내역이 많아질 때 원하는 내역을 빠르게 찾을 수 있게 합니다.

### 3. 트랜잭션(거래) 수정 기능

- 현재 거래 내역은 **추가/삭제만** 가능하고, **수정이 불가능**합니다.
- 금액 오입력, 설명 수정 등의 상황에서 삭제 후 재입력이 필요한 불편함이 있습니다.
- 기존 거래 추가 다이얼로그를 **수정 모드**로 확장하여, 거래 항목 클릭 시 수정할 수 있게 합니다.
- (백엔드 API는 이미 구현되어 있으나, 프론트엔드에서 연결하지 않은 상태)

### 4. 수입원 활성/비활성 토글

- 현재 수입원 카드에 "활성/비활성" 상태가 **텍스트로만 표시**되며, 상태를 변경하는 UI가 없습니다.
- 수입원 카드에 **토글 스위치**를 추가하여 빠르게 활성/비활성 전환이 가능하게 합니다.
- 비활성 수입원은 대시보드 통계에서 제외할지 여부는 추후 결정합니다 (현재는 UI 상태만 관리).

## 2026-02-11: 핵심 기능 기획 및 유용성 강화 논의

사용자(기획자/개발자 페르소나)의 니즈를 반영하여, 단순 기록을 넘어 **'수익성 판단'과 '의사결정'을 돕는 도구**로서의 핵심 기능을 구체화했습니다.

### 1. 부수입원(Income Source)의 유형화 (Classification)

부수입원의 성격에 따라 입력해야 할 데이터와 중요 지표가 다릅니다. 이를 `Type`으로 구분하여 관리합니다.

- **노동 기반 (Labor-based)**: 파트타임, 외주 개발, 컨설팅 등.
  - _핵심_: 투입 시간 대비 수익 (Hourly Rate).
  - _입력_: 근무/작업 시간 필수.
- **사업/자산 기반 (Business/Asset-based)**: 앱/웹 서비스, 블로그(광고), 전자책 등.
  - _핵심_: 투자금(비용) 대비 수익 (ROI), 유지보수 비용.
  - _입력_: 초기 투자금, 정기 지출(서버비 등).
- **금융/투자 (Investment)**: (일단 부수입 관리 범위 내에서는 배제하거나 단순 수익으로 처리).

### 2. 트랜잭션(수익/지출) 모델링의 고도화

단순 `+`, `-`를 넘어선 현실적인 입출금 내역 관리가 필요합니다.

#### A. 수익 (Revenue)

- **불규칙성 대응**: 날짜별 수입 기록 (예: 프리랜서 건별 정산, 애드센스 월별 정산).
- **관련 메타데이터**: 어떤 프로젝트(Source)에서 발생했는지 연결.

#### B. 지출 (Expense) - 복잡성 해결

- **단발성 지출 (One-time)**: 장비 구입, 도메인 1년 결제, 외주비 등. -> 해당 월의 비용으로 처리.
- **정기 지출 (Recurring/Subscription)**: 서버비($5/월), 소프트웨어 구독 등.
  - _아이디어_: `Subscription` 모델을 별도로 두어, 매월 자동으로 지출 내역을 생성하거나, 예상 비용 산정에 활용.

### 3. 유용성 도출을 위한 지표 (Metrics for Utility)

대시보드와 상세 페이지에서 보여줄 핵심 질문들에 대한 답입니다.

- **"이거 계속 하는 게 맞나?" (존속 여부 판단)**
  - **순수익 (Net Profit)**: `총 수익 - 총 지출`.
  - **손익분기점 (BEP)**: 초기 투자금이 회수되는 시점 예측.
- **"내 시간이 얼마나 들어갔지?" (효율성 판단)**
  - **ROE (Return on Effort)**: `순수익 / 총 투입 시간`. (최저시급과 비교 가능).
  - **ROI (Return on Investment)**: `순수익 / 총 지출 비용 * 100`.

### 4. UI/UX 구체화 제안

- **트랜잭션 입력 모달**: "오늘 서버비 나갔음", "오늘 외주비 들어왔음"을 빠르게 입력.
  - 입력 시 `Source` 선택 -> `Type`(수익/지출) 선택 -> 금액/날짜 입력.
- **종합 분석 페이지**:
  - 차트: 월별 수익/지출 막대 그래프.
  - 파이 차트: 나의 수입 포트폴리오 비중 (예: 외주 60%, 앱수익 40%).

## 2026-02-11: 프론트엔드 UI 전면 리디자인 — 엔터프라이즈급 모노크롬 디자인 시스템

기존 프론트엔드가 shadcn/ui 기본 스타일 + 다채로운 색상(파랑, 녹색, 빨강, 보라)으로 구성되어 **진부하고 템플릿 느낌**이 강했습니다. 서비스가 "부수입/재무 관리"라는 진지한 목적을 가지고 있으므로, **신뢰감과 전문성을 전달하는 디자인**이 필수적이라 판단했습니다.

### 1. 컬러 전략: 왜 모노크롬인가

#### 문제 인식

- 수익은 녹색, 지출은 빨간색이라는 **관습적 색상 코딩**은 재무 앱에서 흔히 사용되지만, 시각적으로 산만하고 "가계부 앱" 느낌을 줍니다.
- 수입원 유형별로 파랑/보라/녹색 뱃지를 다르게 칠한 것도 정보 구분에 실질적 도움이 되지 않으면서 화면 복잡도만 높였습니다.

#### 결정

- **흰색 · 검정 · 회색**만 사용하는 완전 모노크롬 팔레트로 전환.
- 수익/지출 구분은 색상이 아닌 **타이포그래피 강약**(foreground vs muted-foreground)과 **부호(+/−)**로 전달.
- 수입원 유형 뱃지는 모두 `bg-neutral-100 text-neutral-600`으로 통일하여, 정보는 텍스트 내용으로만 구분.
- 이 접근은 Stripe Dashboard, Linear, Vercel 등 **실제 엔터프라이즈 SaaS**에서 검증된 패턴.

#### 근거

- 모노크롬 디자인은 **콘텐츠(숫자, 텍스트)에 집중**하게 만들어 "데이터 도구"로서의 성격에 부합.
- 색상이 적을수록 **시각적 위계(hierarchy)**가 명확해져, 사용자가 핵심 정보를 빠르게 파악 가능.
- 접근성(colorblind) 측면에서도 색상 의존도를 낮추는 것이 유리.

### 2. 레이아웃 전략: 사이드바 네비게이션 도입

#### 문제 인식

- 기존에는 단순히 `p-8` 패딩만 적용된 단일 컬럼 레이아웃.
- 페이지 간 이동 동선이 불명확하고, 서비스 정체성을 알 수 없는 구조.

#### 결정

- **고정 사이드바(240px)** + 메인 컨텐츠 영역으로 분리.
- 사이드바에 "부수입 매니저" 브랜드명과 네비게이션 링크(대시보드, 수입원 관리) 배치.
- `react-router-dom`의 `NavLink`로 현재 위치 시각적 표시 (`bg-neutral-100`).

#### 근거

- 데이터 관리 도구는 **여러 섹션을 빈번하게 오가는** 사용 패턴이 있으므로, 항상 보이는 네비게이션이 적합.
- 사이드바는 **서비스의 구조와 규모를 암시**하여, 기능이 적더라도 "제대로 된 서비스" 느낌을 줌.
- 향후 기능 확장(설정, 리포트, 목표 관리 등) 시 자연스럽게 메뉴 항목을 추가 가능.

### 3. 마이크로 인터랙션: Framer Motion 활용 원칙

#### 문제 인식

- 기존 UI는 정적(static)이어서, 데이터가 로드되거나 화면이 전환될 때 생동감이 없음.

#### 결정 및 적용 범위

| 적용 위치                 | 모션 유형                            | 이유                                                  |
| ------------------------- | ------------------------------------ | ----------------------------------------------------- |
| 페이지 전환               | `opacity: 0→1` + `y: 12→0` (0.35s)   | 새 페이지에 "도착했다"는 시각적 피드백                |
| 통계 카드 (StatsCards)    | stagger `delay: i*0.08`              | 3개 카드가 순차적으로 나타나며 데이터에 주목하게 유도 |
| 수익 순위 (SourceRanking) | stagger `delay: i*0.06` + `x: -12→0` | 리스트가 위에서 아래로 흘러내리듯 등장                |
| 수입원 카드 호버          | `whileHover: { y: -2 }`              | 클릭 가능한 요소임을 미세한 부상(lift)으로 전달       |
| 거래 내역 항목            | stagger `delay: i*0.04` + `y: 8→0`   | 긴 리스트도 자연스럽게 로드되는 느낌                  |

#### 원칙

- **체감되되 의식하지 않는 수준**: duration은 모두 0.2~0.4초 이내. 과장된 bounce나 spring은 사용하지 않음.
- **stagger는 짧게**: 항목이 많아질수록 대기 시간이 길어지므로 delay 간격은 0.04~0.08초로 제한.
- **hover 모션은 최소한으로**: `y: -2`만으로 충분. 크게 움직이면 레이아웃이 흔들리는 느낌.

### 4. 타이포그래피: Pretendard 선택

#### 문제 인식

- 브라우저 기본 시스템 폰트는 운영체제마다 다르게 렌더링되어, macOS에서는 괜찮아도 Windows에서는 가독성이 떨어질 수 있음.

#### 결정

- **Pretendard Variable** (CDN) 적용.
- 한글 최적화된 산세리프 폰트로, Inter와 유사한 geometric 특성을 가져 수치 데이터 표시에 적합.
- `-webkit-font-smoothing: antialiased` 적용으로 렌더링 일관성 확보.

### 5. 컴포넌트별 리디자인 포인트

#### 카드 (Card)

- 기존: shadcn 기본 `shadow` → 현재: `shadow-none` + `border border-border`로 변경.
- 호버 시에만 `hover:shadow-sm` 또는 `hover:shadow-md`로 미세한 깊이감.
- **이유**: 그림자 기본값이 강하면 카드가 "떠 있는" 느낌이 과해져 모노크롬 톤과 충돌.

#### 버튼

- CTA(Call-to-Action) 버튼: `bg-foreground text-background` (검정 바탕 흰 글씨).
- 보조 버튼: `variant="outline"` 또는 `variant="ghost"`.
- **이유**: 검정 버튼은 가장 강한 시각적 강조이며, 모노크롬 환경에서 유일한 "주목점" 역할.

#### 빈 상태 (Empty State)

- `border-dashed border-border` + 중앙 정렬 텍스트.
- 산만한 색상 배경(`bg-gray-50`) 대신 투명 배경에 점선 테두리만으로 "여기에 데이터가 들어올 자리"를 시사.

### 6. 라우팅 구조 변경

- 기본 경로(`/`)를 `/income-sources` → `/dashboard`로 변경.
- **이유**: 사용자가 서비스에 진입했을 때 **전체 현황을 먼저 파악**하는 것이 자연스러운 워크플로우. 대시보드에서 개별 수입원으로 drill-down하는 Top-down 탐색 흐름.

### 7. 한글화 원칙

- 기존 영문 혼용("Status", "Active", "All time", "Net Profit" 등)을 모두 한국어로 전환.
- **이유**: 프로젝트 규칙에 한국어 우선이 명시되어 있으며, 타겟 사용자가 한국어 사용자.
- Select 옵션도 enum 값(`FREELANCE`) 그대로가 아닌 `IncomeSourceTypeLabel` 한글 라벨로 표시.
